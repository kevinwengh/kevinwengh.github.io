What Spark Actually Ships to Executors

July 3, 2026

Data PlatformsDistributed Systems

Spark SQL’s whole-stage code generation is often described with a misleading shortcut: “Spark compiles code on the driver and sends it to executors.”

That is close enough to sound right and wrong enough to confuse debugging, performance analysis, and operational decisions.

In Spark 4.1.2, the driver generates Java source for compatible physical-plan subtrees and performs a driver-side Janino compilation for validation and bytecode-size checks. But the driver does not normally send that generated driver class to every executor. The generated source and references travel inside the serializable execution closure, or optionally through a broadcast for large source. Each executor JVM compiles the source on its first relevant cache miss and then reuses the generated class locally.

Planning belongs to the driver

An action begins with the driver. Catalyst parses, analyzes, optimizes, and physically plans the query. Spark then inserts exchanges, row/columnar transitions, and whole-stage codegen wrappers where compatible operators can be fused.

This is the point where it helps to separate three meanings of “stage.” A scheduler stage is split by a shuffle dependency. An adaptive query stage is a materializable fragment that can provide statistics for later planning. A whole-stage codegen stage is a fused subtree of compatible row operators. They overlap, but they are not the same thing.

Code generation creates a program, not a result

The driver’s physical plan becomes an RDD lineage. For a fused row pipeline, WholeStageCodegenExec assembles Java source for a BufferedRowIterator that embodies the work of several operators without allocating an intermediate row or dispatching through a virtual method at every boundary.

The source is valuable because it makes a larger straight-line program available to the JVM. Spark’s Janino path validates it on the driver and checks constraints such as generated bytecode size. If that fails and fallback is allowed, Spark can use a non-whole-stage path for that subtree.

But planning and validation are not partition execution. The driver is constructing the program; it is not running that program over every partition.

Tasks carry execution state to executor JVMs

DAGScheduler cuts the RDD graph at shuffle boundaries, builds a stage binary, and creates partition tasks. The scheduler backend sends a task description to an executor. The executor fetches dependencies, deserializes the task and closure, resolves generated source, and looks up the JVM-local generated-class cache.

On a cache miss, the executor compiles source with Janino. On later tasks in the same executor JVM, the class can be reused. Then HotSpot may JIT-compile hot code further into optimized native instructions.

That layered path matters. A driver-side compile error, an executor classloader problem, a Janino cache miss, and a JVM warm-up effect can all look like “Spark codegen is slow,” but they occur at different points in the lifecycle.

Why this model improves debugging

When a query misbehaves, ask a precise question: is the issue in driver planning, generated-source validation, task serialization, executor-side compilation, partition work, shuffle behavior, or adaptive replanning?

The answer determines where to look. explain and codegen inspection help with the physical plan. Driver logs help with planning and task creation. Executor logs and metrics help with classloading, compilation, and partition execution. Shuffle metrics help with stage boundaries. Runtime statistics explain why AQE may have changed the downstream plan.

The useful mental model is simple: the driver designs and distributes an execution program; executor JVMs realize that program for partitions. Once that distinction is clear, whole-stage codegen becomes less magical and much easier to reason about.
